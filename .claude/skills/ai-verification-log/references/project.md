# PJ単位 検証ログ — 処理フロー＆テンプレート

複数ツール・施策をまとめたプロジェクト記録。施策全体の背景・学び・今後の展開を管理する。

---

## コマンド別処理フロー

### 🟢 init（新規作成）

1. ファイルパスを決定する（指定がなければ `structured/projects/YYYYMMDD_[PJ名].md`）
2. 既存ファイルがある場合は上書きせずユーザーに確認する
3. 以下の雛形テンプレートを書き出す
4. 完了メッセージを出す：`✅ 雛形ファイルを作成しました。パス: [パス]`

### 🔄 update（差分追記）

1. 対象ファイルを読み込む（パス未指定なら `structured/projects/` 内の最新ファイルを候補提示）
2. 入力メモを前処理してセクションに振り分ける（下の「振り分け表」参照）
3. 既存内容は削除せず箇条書きの末尾に追記する
4. プレースホルダー（`（要記入）` `（随時更新）`）はメモが入ったら置き換える
5. ファイル冒頭の `最終更新:` を今日の日付に書き換える
6. ファイル末尾の更新ログに `YYYY/MM/DD | [更新内容の概要]` を追記する
7. ファイルを上書き保存する
8. 完了メッセージ：`✅ ログを更新しました。パス: [パス] / 更新箇所: [セクション名]`

### 🏁 finalize（最終化）

1. ファイルを読み込んで空欄箇所を把握する
2. プロジェクトステータスを `進行中` → `完了` に変更する
3. 期間の終了日を今日の日付に更新する
4. 残っている空欄を `（情報なし）` に統一する
5. 更新ログに `YYYY/MM/DD | 最終化（finalize）` を追記する
6. ファイルを上書き保存する
7. 残っている空欄セクションをユーザーに報告し、Note/SNS用サマリー生成の要否を確認する

| URL・資料・参考記事 | 📚 関連リソース |
| その他・雑多なメモ | ✅ メモ・雑記 |

## publishモード：記事ドラフト自動生成

finalizeモード後、またはユーザーが「記事にして」「Note記事を作って」等で明示的に要求した場合に実行する。
**処理の全体フローは SKILL.md の Step 2-A に従う。** ここではPJ単位の記事テンプレートのみ定義する。

### Note / ブログ記事テンプレート

```markdown
---
title: "[PJ名]の取り組みまとめ：[一言で表した結論]"
status: draft
media: note
created: YYYY-MM-DD
updated: YYYY-MM-DD
source_project: "[元プロジェクトパス]"
published_url: ""
---

# [PJ名]の取り組みまとめ：[一言で表した結論]

## TL;DR
- [結論1]
- [結論2]
- [結論3]

## 背景
[なぜこのプロジェクトを始めたか。読者が共感できる課題感を1〜3文で]

## やったこと
[取り組み内容を時系列で簡潔に]

### 使用した技術・ツール
- [技術/ツール1]
- [技術/ツール2]

## わかったこと

### うまくいったこと
- [Good1]
- [Good2]

### 課題・改善点
- [Bad1]
- [Bad2]

### 意外だった発見
- [検証ログの「学びとナレッジ」から抽出]

## 成果
[定量・定性の成果をまとめて]

## 今後の展開
[次のアクションや展望]

---
#AI #AIツール #[PJ名] #検証 #ナレッジ
```
