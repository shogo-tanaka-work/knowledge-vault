# AI検証ログ管理スキル

検証の「開始 → 途中更新 → 完了 → 記事化」を通じてログファイルをローカルで育てていく、2フォーマット・4コマンド構成のスキル。

---

## Step 0: フォーマットの判断

ユーザーの指示・文脈から、どちらの記録タイプかを判断してから対応するreferenceファイルを読み込む。

### 判断基準

| 指示・文脈の特徴 | フォーマット | 読み込むファイル |
|---|---|---|
| 複数ツール・施策をまとめた記録、プロジェクト全体の振り返り、「PJログ」「プロジェクト記録」 | **PJ単位** | `references/project.md` |
| 特定ツール1つの機能検証、品質・コスト評価、比較検討、「ツール検証」「〇〇を試した」 | **ツール単位** | `references/tool.md` |

### 判断できない場合

どちらか不明な場合はユーザーに確認する：

```
PJ単位（複数ツール・施策をまとめたプロジェクト記録）と
ツール単位（特定のツール1つを深掘り評価）のどちらですか？
```

---

## Step 1: コマンドの確認

フォーマットが決まったら、以下のコマンドモードを確認して対応するreferenceの処理フローに従う。

| ユーザーの指示例 | モード |
|---|---|
| 「ログ作成して」「雛形を作って」 | **init** — 新規ファイル生成 |
| 「ログ更新して＋メモ」「追記して」 | **update** — 差分追記・上書き保存 |
| 「ログ完成させて」「最終化して」 | **finalize** — 整理・ステータス更新 |
| 「記事にして」「Note記事を作って」「ブログにして」 | **publish** — 記事ドラフト自動生成 |

---

## Step 2: referenceファイルを読み込んで処理を実行

判断したフォーマットに応じてreferenceファイルを読み込み、そこに記載されたフォーマット・処理フロー・テンプレートに従って処理を実行する。

- PJ単位 → `references/project.md` を読む
- ツール単位 → `references/tool.md` を読む

---

## Step 2-A: publishモードの処理フロー

`publish` モードは検証ログを元に記事ドラフトを自動生成し、正本ディレクトリへの配置・シンボリックリンク作成・INDEX更新までを一括実行する。

### publishの実行手順

#### 1. 入力ソースの特定
- ユーザーがファイルパスを指定 → そのファイルを読み込む
- 指定なし → `structured/` 配下の最新ファイルを候補提示
- 検証ログ以外（プロジェクトのソースコード、既存ドキュメント等）も入力として受け付ける

#### 2. 出力メディアの判断

| ユーザーの指示 | メディア | 出力先 |
|---|---|---|
| 「Note記事にして」「ブログにして」 | Note | `outputs/note/` |
| 「YouTube台本にして」「動画台本にして」 | YouTube | `outputs/youtube/` |
| 「Udemy教材にして」「講座にして」 | Udemy | `outputs/udemy/` |
| 「SNS投稿にして」「ツイートにして」 | SNS | `outputs/sns/` |
| 「HP記事にして」「Astro記事にして」 | HP | `outputs/hp/` |
| 「全メディアに展開して」 | 全部 | 上記すべて |

#### 3. テーマディレクトリの作成（structured/articles/ に原本を配置）
```bash
THEME="[テーマ名のslug]"  # 例: google-ads-mcp, claude-code-worktree
BASE="structured/articles"
mkdir -p "$BASE/$THEME"
```
- テーマ名は英語kebab-caseで命名
- ユーザーに確認してから作成
- **原本は必ず `structured/articles/` に配置する**（`outputs/` には変換後の成果物を出力）

#### 4. 記事ファイルの作成（フロントマター付き）
`structured/articles/[テーマ名]/` に原本ファイルを作成する。
各ファイルの先頭に以下のフロントマターを必ず付与する:

```yaml
---
title: "記事タイトル"
status: draft
media: note | hp | youtube | udemy | sns
series: "シリーズ名"           # シリーズの場合のみ
series_order: 1               # シリーズの場合のみ
created: YYYY-MM-DD
updated: YYYY-MM-DD
source_project: "MCPServer/mcp-google-ads-remote"  # 元プロジェクト
published_url: ""             # 公開後にURLを記入
---
```

ステータス定義:
| ステータス | 意味 |
|:---:|:---|
| `draft` | 執筆中・初稿 |
| `review` | 推敲中 |
| `ready` | 公開準備完了 |
| `published` | 公開済み |
| `archived` | 廃止・統合済み |

#### 5. 記事本文の生成
- 入力ソースから情報を抽出し、対象メディアのテンプレートに従って本文を生成
- **情報を捏造しない** — ソースにない情報は `（要追記）` で残す
- **機密情報のマスク** — トークン、シークレット、API Key等は `your-xxx-here` 形式に自動マスク

#### 6. シンボリックリンクの作成
元プロジェクトが特定できる場合、プロジェクト側に自動でシンボリックリンクを作成する。
リンク先は `structured/articles/` の原本を直接参照する:
```bash
PROJECT_DIR="[元プロジェクトのパス]"
mkdir -p "$PROJECT_DIR/docs/note"
cd "$PROJECT_DIR/docs/note"
# 相対パスでシンボリックリンクを作成（ポータブルSSDの特性上）
ln -s "[相対パス]/knowledge-vault/structured/articles/$THEME/[ファイル名].md" .
```
- プロジェクト側に既にファイルがある場合はユーザーに確認してから上書き

#### 7. INDEX.md の更新
`outputs/[media]/INDEX.md` に新しい記事エントリを追加する。INDEX.mdが存在しない場合は新規作成する。
リンク先は `structured/articles/` の原本を相対パスで参照する。

INDEX.mdのフォーマット:
```markdown
# [メディア名]記事 ダッシュボード

## 記事一覧

### [シリーズ名/テーマ名]

原本: `structured/articles/[テーマ名]/`

| # | タイトル | ステータス | 更新日 |
|:---:|:---|:---:|:---:|
| 1 | [タイトル](../../structured/articles/テーマ名/ファイル名.md) | `draft` | YYYY-MM-DD |
```

#### 8. 完了報告
```
✅ 記事を作成しました

📁 原本: knowledge-vault/structured/articles/[テーマ名]/
   - [ファイル一覧]
🔗 シンボリックリンク: [プロジェクトパス]/docs/note/
   - [リンク一覧]
📋 outputs/[media]/INDEX.md を更新しました

次のステップ:
- 記事の内容を確認・推敲してください
- 完了したら status を review → ready → published に更新してください
```

## 共通ルール（全モード・全フォーマット共通）

1. **情報を捏造しない** — メモにない情報は `（情報なし）` で残す
2. **既存内容を消さない** — updateモードでは追記のみ。既存の箇条書きは保持する
3. **プレースホルダーを揃える** — 情報が入ったら置き換え。なければ `（情報なし）` に統一
4. **更新ログを必ず記録する** — ファイル末尾の更新ログテーブルに毎回追記する
5. **ファイルパスを明示する** — 操作するファイルのパスを毎回ユーザーに伝える

### Slack転記の前処理ルール（全フォーマット共通）

`名前: 発言内容` または `[時刻] 名前\n発言内容` 形式を検出したら：
1. 発言者名・タイムスタンプを除去
2. 発言内容だけを抽出してメモとして処理

### 音声メモの前処理ルール（全フォーマット共通）

音声文字起こしテキスト（Amical等）を検出したら：
1. フィラーワード（えー、あのー、まあ、なんか、ちょっと、えっと）を除去
2. 口語表現を書き言葉に変換（〜じゃないですか→〜である、〜なんですけど→〜だが、〜って感じ→〜という印象）
3. 繰り返し・言い直しを整理して最終的な意図のみ残す
4. 内容をメモ振り分け表に従ってセクションに振り分ける

### 推奨ディレクトリ構成

```

├── structured/                    ← 全コンテンツの原本
│   ├── tools/                     ← ツール単位の検証ログ
│   │   └── 20260401_n8n-ai-agent-node.md
│   ├── projects/                  ← PJ単位の検証ログ
│   │   └── 20260331_n8n-automation.md
│   └── articles/                  ← 記事原稿（テーマ別サブディレクトリ）
│       └── google-ads-mcp/
│           ├── 01-motivation-and-auth.md
│           └── 02-local-mcp-server.md
├── outputs/                       ← メディア別変換済み成果物
│   ├── note/
│   │   └── INDEX.md               ← ダッシュボード（ステータス管理）
│   ├── youtube/
│   ├── hp/
│   ├── udemy/
│   └── sns/
└── raw/                           ← 音声メモ等の生データ
```

**データフロー:**
```
structured/ (原本)  → [publishスキル] → outputs/ (メディア別成果物/INDEX)
                                              ↓ シンボリックリンク
                                    プロジェクト/docs/note/ (参照)
```

### CLAUDE.md への推奨設定

```markdown
## 検証ログの運用ルール
- スキル: ai-verification-log を使用すること
- PJ単位の保存先: structured/projects/
- ツール単位の保存先: structured/tools/
- ファイル名: YYYYMMDD_[名称].md
- initは検証開始当日に実行する
- updateは都度手動で指示する（自動更新しない）
- finalizeは検証完了後に実行する
```
