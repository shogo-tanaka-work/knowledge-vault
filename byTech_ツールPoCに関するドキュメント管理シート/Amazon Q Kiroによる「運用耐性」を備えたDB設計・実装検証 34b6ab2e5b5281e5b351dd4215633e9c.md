# Amazon Q/Kiroによる「運用耐性」を備えたDB設計・実装検証

ステータス: 完了

**カテゴリ:** ツールPoC（システム開発・AIエージェント）

**期間:** 2025/08/30 - 2025/09/01

**プロジェクトステータス:** 完了（開発フローの再定義）

### 1. 背景と目的

- **開始理由:** 前回のKiro検証では「骨組み」の速さは確認できたが、実際のプロダクト運用に耐えうる「詳細設計（DB正規化、非機能要件）」がどの程度網羅できるかを検証するため。
- **解決したい課題:** ログアウト機能、操作ログ、パスワード再設定など、AIが「指示なしでは落としがちな必須機能」の特定と、バックエンド（Supabase）連携の最適化。

### 2. 取り組み内容

- **実施施策:** 1. チャッピー（ChatGPT）で機能・非機能要件を詳細化しKiroに投入。
    1. バックエンドをSupabaseに統一し、UUIDによるユーザー管理を実装。
    2. パフォーマンス基準を提示し、実務レベルのレスポンスを要求。
- **使用ツール・技術:** Amazon Q/Kiro, Cursor (Claude 3.5 Sonnet / Grok-1), Supabase

### 3. 進捗と成果

- **定性的な成果:** Kiro単体では「操作ログ」「ログアウト」等の運用必須機能が漏れる傾向を特定。これらを事前に要件定義書に含めるプロセスの重要性を確認。
- **技術的な発見:** 正規化やインデックス設定など、DB設計の深い部分は実装前に「表形式の設計書」としてAIに出力させ、人間が目視チェックを行う工程が最適解であると結論。

### 4. 学びとナレッジ

- **うまくいったこと（Good）:** Kiroの`specs`をベースに、Cursorで「バイブコーディング（対話型実装）」を行うことで、実装速度を維持しつつ柔軟な手直しが可能になった。
- **改善ポイント（Improve）:** ファイル構造（HooksとAPIの切り分け等）がプロジェクト標準とズレる場合があるため、ディレクトリ構成も事前にKiroへ指示すべき。

### 5. 課題と対応

- **発生した課題:** 非エンジニアが扱う場合、画面上の数値が「固定値」か「DB変数」かの判別が難しく、修正指示が迷走するリスクがある。
- **対応方法:** バックエンドの基礎知識（初学者レベル）を前提としたプロンプト管理、またはAIに「現在どのデータを参照しているか」をコメント出力させる手法を検討。

### 6. 今後の展開

- **次のアクション:** 結合テストにおける「テストケース生成」と「ダミーデータ作成」もAIで自動化し、パフォーマンス検証まで一気通貫で行うフローを構築する。
