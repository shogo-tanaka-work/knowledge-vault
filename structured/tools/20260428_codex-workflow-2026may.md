## 1. 背景と検証目的

### なぜこの検証を行うのか

長年 Claude Code を主力ツールとして使ってきた。Claude Design のリリース当日も触れ、Anthropic 製品を優先的に採用してきた立場。
しかし 2026 年 4 月 16 日の「Codex for (almost) everything」発表後、Codex の立ち位置が根本的に変わったと判断したため、現場ワークフローへの影響を徹底検証した。

### 解決したい課題

- コーディング・LP制作・デザインモックのそれぞれで「一番生産性が高いツール選択」を確定させたい
- 法人研修クライアントへの AI ツール推薦を根拠ある形で行いたい
- 「なんとなく使ってる」ツール選択から抜け出す

### 期待される効果

- 1日あたりの制作量 2〜3倍
- 「作業」カテゴリの仕事をAIに移譲し、「判断・創造・対話」に集中できる時間の確保

## 3. 主要機能別 検証結果

### 3-1. 並列エージェント + Git Worktrees

**検証シナリオ**: 朝一番に以下 7 タスクを同時投入

```
・TypeScript エラー修正
・Webhook エンドポイントのスキーマ更新
・管理画面のエラーバウンダリ追加
・レガシー auth middleware 移行
・LP 追記セクション実装
・note 記事サムネ画像生成＆配置
・Gmail 未返信問い合わせ抽出＆返信ドラフト作成
```

**結果**:
- 各エージェントが独立した Git Worktree で稼働 → 衝突なし ✅
- 約 20 分で 4 タスクが承認待ちに到達 ✅
- 設計方針が固まっていない初期段階では「3案並列作成→比較して決定」の使い方が有効 ✅
- 待ち時間ゼロ。1タスクが詰まっても全体は止まらない ✅

**WorkOS 社の報告**: 朝のメンテナンスタスクの 30〜40% 削減を実証済み

---

### 3-2. In-appブラウザ + ローカルサーバー連携

**検証シナリオ**: Vite + React のフロントエンドプロジェクトで LP 微調整

**操作フロー**:
1. Codex アプリ内のブラウザでローカルサーバー（localhost:5173）を開く
2. CTAボタンに「もう少し大きく・赤系に」とコメントを書き込む
3. Codex が該当コンポーネントを自動で特定して修正
4. ホットリロードで即反映

**結果**:
- コードエディタへの往復が不要に ✅
- 「ブラウザ → コード → ブラウザ」サイクルが 1 ウィンドウ内で完結 ✅
- CSS の上書き場所を探す調査工数がほぼゼロ ✅
- ビューポート切替（PC / タブレット / スマホ）もアプリ内で完結 ✅
- LP 制作工数が体感で 1/5 に削減 ✅

---

### 3-3. gpt-image-1.5 画像生成統合

**検証シナリオ**: LP ヒーローセクション用画像の生成→配置→CSSスタイリング

**操作フロー（従来）**:
Midjourney → プロンプト工夫 → ダウンロード → エディタでパス指定 → 表示確認（5ステップ）

**操作フロー（Codex）**:
「このセクションに画像生成して配置して」 → gpt-image-1.5 で生成 → 適切なパスへ配置 → background-image 設定まで完結（1アクション）

**結果**:
- 参考スクリーンショットを渡せば「雰囲気を合わせる」編集も可能 ✅
- コード・画像・スクショが同一画面で循環 ✅
- FV 1 本 = 1 時間以内（従来の半日比） ✅

## 4. ユースケース別 実証結果

### 4-1. コーディング（TypeScript / React）

**実証内容**: 法人研修クライアント向けのプロポーザル資料（HTMLスライド 20ページ相当）

```
エージェントA: TailwindCSS + アニメーション付きHTMLスライド実装
エージェントB: Mermaid形式ロードマップ図 + gpt-image-1.5で表紙ビジュアル生成
エージェントC: ROI試算Excelシート（グラフ込み）生成
```

**結果**:
- 3エージェント並列 → 約 2 時間でほぼ完成（従来の丸2日比） ✅
- 各エージェント独立で詰まっても全体が止まらない ✅
- PR が上がった時点での修正コストが体感 3 割減（GPT-5.5効果） ✅

---

### 4-2. LP・Webサイト制作

**実証内容**: 新規 LP（FV + 4セクション + レスポンシブ対応）の制作

- Figma カンプ → Codex（画像解析） → ベース実装 → ブラウザ内コメント微調整
- FV 1 本: **1時間以内**（従来の半日比）
- LP 全体: **1日**（従来の3日比）
- 複数ページ（サンクス / プライバシー等）はページ別にエージェントを分割して並列処理

**クライアントへの提案スピード革命**:
- 従来: A案・B案を両方作り込むのに 2〜3日
- 現在: 同じ午後中に両方の動くプロトタイプを提示可能 → 案件獲得率に直結

---

### 4-3. デザインモック・ラフ案

**実証内容**: AIエージェント導入ラフ案ドキュメント（概要図 + ユーザーフロー図 + 画面モック 10枚）

```
1. 概念図を gpt-image-1.5 で生成 → 修正指示 → HTMLスライドに埋め込み
2. ユーザーフロー図を Mermaid 形式で生成 → ブラウザ内でレンダリング確認
3. 画面モックを React コンポーネントとして実装 → 「クリックできる動くモック」として納品
```

**所要時間**: 半日（従来の1日半比）

**クライアント反応**: 静的画像ではなく「動くモック」を見てもらえることで提案の説得力が段違い

## 5. 比較分析

### Claude Code vs Codex（2026/04/28 時点）

| 観点 | Claude Code | Codex |
|---|---|---|
| コード単体性能 | ◎ SWE-bench 80.8% | ◎ 同等水準 |
| ターミナル統合 | ◎ 主戦場 | ○ CLI あり |
| In-appブラウザ | ✗ なし | ◎ あり |
| 画像生成統合 | ✗ なし | ◎ gpt-image-1.5 |
| 並列エージェント | △ 限定的 | ◎ Git Worktrees で完全並列 |
| プラグイン数 | △ MCP中心 | ◎ 90以上（公式サポート） |
| インターフェース | ターミナル | デスクトップアプリ |
| 非エンジニアの利用 | ✗ 困難 | ◎ ハードルが低い |
| コンテキスト長 | ◎ 100万トークン | △ 400K（Desktopは制限あり） |

**Claude Code が優位な場面**:
- ターミナルでのシェル操作が主体
- 大規模コードベースの理解（100万トークン活用）
- 長文ドキュメント作成・複雑な推論

---

### Claude Design vs Codex（デザインモック作成での比較）

| 観点 | Claude Design | Codex |
|---|---|---|
| デザイン出力品質 | ◎ Opus 4.7 のビジョン活用 | ○ gpt-image-1.5 |
| 実装への引き継ぎ | △ 別ツール（Claude Code）に渡す | ◎ 同一ツール内で完結 |
| ワークフロー統合度 | △ デザインフェーズのみ | ◎ コード・画像・ブラウザが1画面 |
| エクスポート先 | Canva / PPTX / HTML | コードとして直接出力 |
| 非デザイナー向け | ◎ 橋渡しとして最強 | ○ 実装者が使うツール |

**Claude Design の優位な場面**:
- 非エンジニアがデザインを起点にしたい場合
- Canva / PPTX への出力が必要な場合
- デザイン専門フェーズ（実装者が別チームの場合）

**Codex が圧倒的な場面**:
- デザイン→実装のサイクルを繰り返す現場
- 1人または少人数で「デザインも実装もやる」ケース

## 8. 学びとナレッジ

### 発見したこと

- Codex の本質は「コード生成ツール」ではなく「制作ワークフローの統合プラットフォーム」
- プロダクト戦略の差が体験差に直結している（OpenAI: スーパーアプリ統合思想 vs Anthropic: ツール特化戦略）
- 並列試行が「設計品質」を上げる副作用がある（頭の中だけでは気づけない選択肢が実装してから見えてくる）
- 非エンジニア層への普及のボトルネックは「ターミナルというインターフェース」だった

### うまくいったこと（Good）

- 朝のタスクキュー仕込み → 20分で 3〜4 タスク完了の体験
- In-appブラウザコメント修正でディスプレイ往復が消滅
- 動くモックでクライアントへの提案説得力が段違いに向上
- AGENTS.md に規約を書いておくことで修正指示の繰り返しが減少

### うまくいかなかったこと（Bad）

- コンテキストウィンドウが 400K（Claude Code の 100 万比）に制限されているため大規模コードベースでは不利
- Codex Desktop の Hooks はまだ未実装（v26.422 時点）→ CLI 版は対応済み
- GPT-5.5 リリース後に config.toml での 1M 設定が機能しなくなった

### Tips・ベストプラクティス

- 指示は日本語でOK（GPT-5.5 の日本語理解は十分）
- AGENTS.md のコーディング規約が充実しているほど修正コストが下がる
- 「意図的に使わない日」を週1回設けることで Codex の価値を客観視できる
- 画像生成は「1発で完璧」を求めず、Codex 内で数回修正するフローに慣れる

### 総合評価

```
コーディング領域       : ★★★★★（並列 + GPT-5.5 で圧倒的）
LP・Webサイト制作      : ★★★★★（In-appブラウザが決定打）
デザインモック・ラフ案  : ★★★★★（画像生成との統合で完結）
長文ドキュメント・推論  : ★★★☆☆（Claude のほうが依然強い）
非エンジニア向け普及    : ★★★★★（デスクトップアプリが決め手）
```

### 判定

- [x] **採用確定**（コーディング・LP・デザインモック領域）
- [x] Claude Code との**住み分け運用**（大規模コードベース・長文推論は Claude 継続）
- [x] 法人研修クライアントへの**推薦ツールを Codex に変更**
