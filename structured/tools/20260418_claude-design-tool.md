## 3. 検証内容：プロンプト壁打ちログ

### テーマ：初期プロトタイプ生成

---

#### ターン1：サイトリデザインの起点

**プロンプト**
```
こちらのサイトのデザインを刷新したいです、アイデア出してサンプル作りたいです。
Hi-fi design
Interactive prototype
shogo-tanaka-HP
```

**Claudeの質問 & ユーザー回答**

Claudeが方向性を決めるために質問を投げ、ユーザーが一括回答した形式。

| 質問項目 | 回答 |
|---|---|
| トーン | 洗練・ミニマル（ライト系） |
| カラー方向性 | 全く新しい配色にしたい |
| ターゲット | 企業（開発案件の受注） |
| ヒーロースタイル | アニメーション・動きのあるヒーロー |
| レイアウト | シングルページ（スクロールで全セクション） |
| スコープ | 全ページ |
| バリエーション数 | 5パターン以上 |
| バリエーションの軸 | レイアウト・構成 / インタラクション・アニメーション |

**Claudeの実行プラン（自律的に提示）**
1. デザインシステム：新カラーパレット×5・タイポグラフィ・スペーシング
2. メインHTML + コンポーネントファイルを5レイアウトバリエーションで構築
3. ヒーローセクション：5種のアニメーションヒーローレイアウト
4. 全セクション：Stats / Services / Portfolio / Skills / Contact / Footer
5. Tweaksパネルでバリエーション切り替え
6. 検証 & 納品

**生成されたバリエーション**

ヒーローレイアウト5種（Tweaksで切り替え可能）：
- A: 左寄せテキスト + 浮遊ジオメトリックシェイプ
- B: 中央配置 + タイピングアニメーション
- C: 左右分割 + モザイクグリッド（初期表示）
- D: 超大型タイポグラフィ + アクセントライン
- E: ミニマル縦ライン + スタッツカード

カラースキーム5種：
- Slate × Sienna（温かみのある琥珀系）
- Mono × Indigo（クールなインディゴ）
- Navy × Terracotta（テラコッタ）
- Graphite × Blue（クラシックブルー）
- Warm × Olive（落ち着いたオリーブ）

**気づき**
- 「Hi-fi design / Interactive prototype」というキーワードだけで、質の高いプロトタイプを前提に動いてくれた
- 質問→一括回答という形式でも、全項目を正確に解釈して実行プランを自律的に組み立てた
- 5レイアウト×5カラーの25通りを一度に生成する実行力はV0と比べて段違い

### テーマ：エントランスアニメーション実装（デバッグ）

---

#### ターン2：JSステートからCSSアニメーションへのデバッグ

**経緯**
ユーザーの明示的なプロンプトなし。Claude が自律的にアニメーション実装の問題を検知し、複数回リライトを行ったターン。

**Claudeの試行フロー**

1. **第1試行**: JSステートベースのエントランスエフェクト実装
   → Babel環境でJSステートが動作しないことが判明

2. **第2試行**: CSSクラスベースへの書き換え（`loaded` クラストグル方式）
   → 同様に動作せず

3. **第3試行**: pure CSS `animation-delay` 方式に完全書き換え
   → `fork_verifier_agent` によるスクリーンショット検証で動作確認できず

4. **最終解**: ヒーローコンテンツをデフォルト表示（`opacity: 1`）にし、CSSアニメーションはプログレッシブエンハンスメントとして残す形で解決

**Claudeのコメント（意訳）**
> アニメーションは通常ブラウザでは動作するが、検証環境のスクリーンショットでは動かない。ヒーローコンテンツを即座に表示し、アニメーションはプログレッシブエンハンスメントとして扱う方針に変更。

**この時点でのUI完成状態**
- Tweaksパネルから5種のヒーローレイアウト・5種のカラースキームを切り替え可能な状態

**気づき**
- ユーザーが何も言わなくても、自律的に問題を検知して複数回リライトを試みる挙動が確認できた
- `fork_verifier_agent` という内部エージェントがスクリーンショット検証を担っており、Claude Design Tool が単体のLLMではなくエージェント構成になっていることが示唆された
- 環境依存の問題（Babel・スクリーンショット検証環境）でフォールバック設計を選択する判断は適切

### テーマ：慣性スクロール & パララックス実装

---

#### ターン3：初期要件の投入

**プロンプト**
```
慣性スクロールとりいれれますか？
簡単で良いのでFVあたりにパララックスの要素を入れてみたいです。
```

**Claudeの対応**
- lerpベースの慣性スクロールドライバーを実装（`useParallax()` フックとして切り出し）
- ヒーロー各バリアント（HeroA/B等）に背景デコレーション（円形・矩形）を追加
- パララックス速度を3段階（slow / mid / fast）で設定
- `App` 側で `useParallax()` を呼び出しCSS変数 `--scroll-y` で全体に伝播

**気づき**
- 要件が「簡単で良い」という抽象的な指示でも、実装レベルを適切に判断して出してきた
- lerpという実装手法を明示せずとも選択してきた点は好印象

---

#### ターン4：強度・視認性の問題を修正

**プロンプト**
```
慣性スクロールの強度をややあげてください
パララックスの表示がカード形式の要素と重なっててわかりにくいです。また動きの量が小さくわかりにくいです。
```

**Claudeの対応**
- lerp係数を調整して慣性を強化
- パララックスの移動量を2〜3倍に拡大（slow: -0.4, mid: -0.22, fast: 0.3）
- デコレーション形状をカード風のボックスから `radial-gradient` の大型ソフトグロー（300〜450px）に変更
- グローを画面端（コンテンツ外）に配置し重なりを解消

**気づき**
- 「カード形式の要素と重なってわかりにくい」という問題報告に対して、位置だけでなく形状そのものを変えて根本解決してきた
- デザイン的な判断（グロー表現に切り替える）を自律的に行っている点が良い

---

#### ターン5：別パターンへの切り替え試行

**プロンプト**
```
慣性スクロール、別パターンを試したい
もっとヘビーにしてスクロールしても今より動きの量を抑えたい
パララックスの要素である円形を左側に配置して
```

**Claudeの対応**
- lerp係数を 0.025 まで下げ、停止後の余韻を強調（より重い追従感）
- パララックス移動量を約4割減（slow: -0.25, mid: -0.14, fast: 0.18）
- 全ヒーローバリアントの円形グローを左側に集約配置

**気づき**
- 「ヘビーにしつつ動きの量は抑える」という一見矛盾する要件を正しく解釈できていた（慣性の重さとパララックス移動量は独立したパラメータ）
- 複数の変更を1ターンでまとめて処理してくれる

---

#### ターン6：配置の微調整

**プロンプト**
```
円形グロー　開始位置をもう少し下にして
```

**Claudeの対応**
- 各ヒーローの円形グロー `top` 値を 15〜20% 付近に調整

**気づき**
- 短い指示でも文脈（左側に配置した円形グローのこと）を正確に維持している
- 1行の微調整指示でも見当違いな変更はしない

---

#### ターン7：移動量の再強化

**プロンプト**
```
パララックス移動量　もっと増やして
```

**Claudeの対応**
- 移動量を前回から約2倍に拡大
  - slow: -0.55
  - mid: -0.35
  - fast: 0.45

**気づき**
- ターン5で抑えた値からターン7で再度増やすという反復調整に素直に追従
- 数値の履歴を保持してスケール感を維持している

## 4. 検証結果・評価

### 良かった点

#### V0の代替として十分
- Figmaデザインの素案をアイデアから素早くプロトタイプに落とし込める
- PMやノンデザイナーがデザイン専門用語なしに「こういうのを作りたい」から形にできる
- 60点クオリティのプロトタイプをかなりのスピードで生成できる

#### 対話的な調整体験が良い
- 「もっとヘビーに」「左側に配置して」等の自然言語でUIを詰められる
- 壁打ちのループが非常にスムーズ
- パターン変更・細部調整を複数ターンで反復できる

### 課題・懸念点

#### コードがベタ書きで出力される
- 単一HTMLやJSXのベタ書きコードが出力される
- コンポーネント分割・モジュール化の設計がなく、そのままサーバーにデプロイするには手が必要
- 本番投入前に構造整理・リファクタリングが必要

#### デザイナー完全代替には至らない
- 仕上がりの品質は「60点のプロトタイプ」レベル
- 高品質なデザインシステム・ブランドガイドライン整合には人手が必要
- デザイナーが「最初に出てきたものを調整する」フェーズは残る

## 5. 学んだこと・知見

### デザイナーの仕事の変化
- Figma/Adobe XDでポチポチ作る作業フェーズは縮小していく
- 「AIにどう指示してデザインを仕上げるか」のプロンプト設計・仕組み整備が主業務になる
- AIの出力を評価・修正できる「デザインの目」は引き続き必要

### ノンデザイナー・PMへの示唆
- デザイン専門知識は不要になるが「良いデザインかどうかの判断軸」は必要
- エンジニアも含め、デザイントレンドの概要把握（例: ベントーデザイン、慣性スクロール、パララックス速度感）が求められる
- 専門用語を知らなくても「なんとなくのイメージを言語化できる」レベルで十分

### Figma/V0との比較感覚
| ツール | 位置づけ |
|---|---|
| V0 | ほぼ不要になる。ClaudeのUIプロト生成で代替可能 |
| Figma | ハイクオリティ仕上げ・デザインシステム管理には引き続き必要 |
| Claude Design | プロトタイプ・素案・PoC用途で最速 |
