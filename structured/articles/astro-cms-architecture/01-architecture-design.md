# Astro × CMS アーキテクチャ設計論 — Content Collectionsから構成選定まで

> Astroでサイトを作るとき、「コンテンツ管理をどうするか」は避けて通れない設計判断。Content Collectionsの仕組みを理解した上で、チーム構成・コスト・リアルタイム性の3軸からアーキテクチャを選定する方法を整理した。

---

## Content Collections とは

Astroが提供する、コンテンツファイルを**型安全に管理・取得するための仕組み**。

- MarkdownやMDXファイルをブログ記事として扱うのが基本的なユースケース
- Zodによるスキーマ定義でフロントマターの型チェック・バリデーションが効く
- `getCollection()` / `getEntry()` などの専用APIでコンテンツを取得する
- ローカルファイルだけでなく**外部CMS・APIからのデータも同じAPIで扱える**のが重要な特徴

ここが単なる「Markdownブログ機能」と違うポイント。外部データソースも統一的に扱えるから、CMS連携のアーキテクチャ選定に直接関わってくる。

---

## コレクションの2種類

Astroのコレクションには「ビルド時」と「ライブ」の2種類がある。

| | ビルド時コレクション | ライブコレクション |
|---|---|---|
| データ取得タイミング | ビルド時 | リクエスト時 |
| 設定ファイル | `src/content.config.ts` | `src/live.config.ts` |
| 主なAPI | `getCollection()` / `getEntry()` | `getLiveCollection()` / `getLiveEntry()` |
| パフォーマンス | 高い（静的HTML） | リクエストごとに取得 |
| 即時反映 | 不可（ビルドが必要） | 可能 |
| MDX・画像最適化 | 対応 | 非対応 |
| 向いているケース | ブログ・ドキュメント・LP | 在庫・株価・リアルタイムデータ |

**基本方針：ビルド時コレクションを優先し、リアルタイム性が必要な場合のみライブコレクションを使う。**

大半のWebサイトはビルド時コレクションで十分。ライブコレクションはパフォーマンスとインフラコストのトレードオフを伴うので、本当に必要なケースに限定する。

---

## チーム・状況別アーキテクチャ選定

「どの構成がベストか」はチーム構成とプロジェクト要件で決まる。5つのパターンを整理した。

### パターンA：エンジニアのみ / Astroを扱える人だけのチーム

```
MDファイル（Git管理） → Astro → 静的HTML → VPS公開
```

- CMSは不要、Astroだけで完結
- 記事はMarkdownで書いてGitにコミットするだけ
- コストが最も低い
- **非エンジニアには不向き**

最もシンプルで、最もコストがかからない構成。個人ブログや技術ドキュメントサイトなら、まずこれを検討する。

### パターンB：非エンジニアの編集者がいる / コストを抑えたい（推奨）

```
WordPress（管理画面専用・外部非公開）
    ↓ 公開ボタン → Webhook発火
GitHub Actions等でAstroビルド実行
    ↓ WordPressのREST APIから記事取得
VPSにデプロイ → 一般ユーザが閲覧
```

- 編集者はWordPressの管理画面だけ触ればよい
- WordPressのライセンスコストなし（サーバ代のみ）
- 公開から反映まで数分のタイムラグあり（運用ルールとして許容する）
- **最もバランスが良い現実解**

### パターンC：Gitベース GUI（Decap CMS / Tina等）

```
編集者がブラウザ上の専用GUIで記事作成
    ↓ 実態はGitへのコミット
Push → 自動ビルドトリガー → Astroビルド → デプロイ
```

- 追加のCMSサーバが不要でコスト安
- GitHubアカウント管理が必要
- WordPress未経験の編集者向け

### パターンD：有料ヘッドレスCMS（Contentful / Sanity等）

- 専用の編集GUIが洗練されており使いやすい
- エントリ数・APIコール数で課金されるためコストが膨らみやすい
- 小〜中規模では過剰投資になりがち

### パターンE：ライブコレクション × SSR（即時反映が必要な場合）

```
CMS → AstroサーバがリクエストごとにAPIを叩く → 即時反映
```

- VPS上でNode.jsサーバを常時稼働させる必要がある
- インフラ管理コスト増
- 在庫・価格など頻繁に変わるデータ向け

---

## ヘッドレスCMS構成の役割分担（パターンBの詳細）

パターンBが「現実解」として推奨される理由を、役割分担の観点から深掘りする。

```
【サイトA（WordPress）】             【本番サイト（Astro）】
管理画面専用・外部非公開               一般ユーザ向け公開サイト

役割：記事データの置き場所             役割：表示・デザインすべて担う
- 記事作成・編集                      - トップ・プロフィール等の静的ページ
- カテゴリ・タグ管理                   - ブログ記事ページ（WPのデータを使用）
- 画像アップロード                     - 統一されたデザイン・UX
- 公開ボタンを押す ──Webhook──→ Astroビルドトリガー
```

**WordPressは「裏方のデータ管理ツール」、Astroは「それを使って自由に表現するフロントエンド」。**

この分離が重要。WordPressのテーマやプラグインに縛られず、Astro側で完全にデザインをコントロールできる。WordPressはあくまで「編集者が使い慣れたCMS管理画面」として活用する。

---

## WordPressの公開ボタンの挙動

ここはよくある誤解ポイント。

- 公開ボタンを押すと**WordPressとしての公開とWebhook発火が同時に起きる**
- 「どちらか一方」を選ぶ操作は標準機能にはない
- WordPressのフロントエンドURLを外部公開しないことで、実質的に「Astroサイトだけが見える」状態を作る

### WordPressを外部に見せない方法

- Basic認証をかける
- VPSのプライベートネットワークに閉じてAstroからのAPIアクセスのみ許可する

### Webhookの設定

Webhookの発火にはプラグインが必要（WP Webhooksなど）。「記事ステータスがpublishになったら指定URLへPOSTする」という設定を入れるだけでよい。

---

## 画像の扱いに関する注意点

ヘッドレス構成でWordPressを使う場合、記事内の画像はWordPressサーバに置かれたままになる。

| 規模 | 方針 |
|---|---|
| 小規模 | WordPressの画像URLを直接参照しても問題ない |
| 本格運用 | Cloudinary・S3等の外部ストレージに逃がすプラグインを使う |
| パフォーマンス重視 | Astroビルド時に画像をダウンロードしてAstro側で画像最適化する |

画像の扱いは後回しにしがちだが、サイト規模が大きくなると確実にボトルネックになる。初期設計の段階で方針を決めておくのが望ましい。

## 選定フローチャート

最終的な意思決定は、以下の3つの問いで整理できる。

```
編集者にエンジニア以外がいるか？
    ├─ No  → Astroのみ（MDファイル＋Git）【パターンA】
    └─ Yes
          ↓
       即時反映が必要か？
          ├─ Yes → ライブコレクション＋SSR【パターンE】
          │         （インフラコスト増を許容）
          └─ No（数分のタイムラグを許容）
                ↓
             コストを抑えたいか？
                ├─ Yes → WordPress（ヘッドレス）＋ Webhook＋ Astro静的ビルド【パターンB】
                │         ← 現実解
                └─ UX重視 → 有料ヘッドレスCMS＋ Astro【パターンD】
```

**設計の基本は「チームの体制・コスト・リアルタイム性」の3軸で判断すること。**

多くのケースでは、パターンAかパターンBに落ち着く。迷ったらシンプルな方を選び、必要になった時点でスケールアップする方が失敗しにくい。
