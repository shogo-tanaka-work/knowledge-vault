# Threads投稿案：DifyなしのLINE簡易RAG 気づきカスケード

5本連投を想定。新着順アルゴリズムを狙い、短時間に投下する。
各投稿は会話体・400〜500字以内。投稿1は単独でも成立する形にする。

---

## 投稿1（気づきベース）

```
LINE公式アカウントのFAQ対応って、同じ質問に何度も答えてる感じが地味にしんどいんだよね。料金は、納期は、キャンセルは…って。

それをAIに一次対応させようとすると、だいたい「また新しい管理ツールが増える」になりがち。今回はそこを避けて、運用する人が触るのはGoogleスプレッドシート1枚だけ、っていう構成で組んでみた。

質問の意味をAIにベクトルで捉えさせて、近いFAQを探して返す。いわゆる簡易RAGをスプレッドシートの上で動かしてる感じ。
```

---

## 投稿2（方針転換した話）

```
最初はDify（AIワークフローのツール）を間に挟む設計だったんだけど、途中でやめた。

理由は2つ。Dify経由だと回答が「Webの画面に飛ばす」形になりやすくて、LINEのトーク内で会話が完結する普通のチャット体験から外れちゃう。あと、Difyのノード設定って作った本人はいいけど、引き継ぐ運用担当者には「もう一個覚えること」になる。

可視化ツールの便利さより、引き継ぎやすさを取った。構成要素をLINE / GAS / スプレッドシート / OpenAIの4つだけに絞ったら、見る場所が一気に減って気持ちよかった。
```

---

## 投稿3（詰まったところ）

```
やってて一番「うっ」となったのが、GASの制約。

LINEのWebhookって、本物のリクエストか検証する署名の仕組みがあるんだけど、これHTTPヘッダを使う。で、GASのdoPostって本文は読めるのにヘッダが読めない。つまりLINE公式の署名検証がそもそも実装できない。

仕方なくURLに秘密のトークンを付ける簡易認証で妥協した。ここは正直、妥協点。署名検証のコード自体は「使わないけど残す」状態にして、将来別のサーバーに移したら即流用できるようにしてある。
```

## 投稿4（次やりたいこと）

```
今のところ、LINE実機から送って「登録済みの質問には正しく回答／無関係な質問にはお断り文言」までは動いてる。E2Eは通った。

ただ、本番はここから。回答精度をちゃんと数字で測るフェーズが残ってる。

正直、簡易RAGを「作る」のはそんなに大変じゃなかった。難しいのは、元のマニュアルをどの粒度のQ&Aに切り分けるか。質問の切り方が雑だと、どんなに検索の仕組みが良くても精度は出ない。ここが次の宿題。
```

## 運用メモ

- 新着順アルゴリズムを狙い、5本を短時間で連投する
- 各投稿は400〜500字以内。投稿1は単独でも刺さる形にする
- 絵文字は控えめ（1投稿につき1個まで、または無し）
- 失敗・妥協を素直に書く（投稿3）。煽り表現は避ける
- 進行中の検証のため「精度が高い」等の断定はしない
