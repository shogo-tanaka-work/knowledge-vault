# noteのトレンド記事を毎朝AIが届けてくれる仕組みを設計した話

## TL;DR

- noteには公式APIがないが、非公式API v3 でスキ数・コメント数・フォロワー数などを取得できる
- スコアリングは「バズ速度」と「エンゲージメント率」の2軸で設計すると、有名クリエイターの普通の記事と無名クリエイターの良記事を両方拾える
- 定期実行基盤はGitHub Actions一択。Cloudflare Workers・Vercel・GASはこの用途では実行時間がネック
- Claude Haikuでテーマ適合性を判定して、Sonnetで通知文を生成。月180円程度で回せる見込み

---

## なぜ作ろうと思ったのか

「生成AIの組織導入」「経営×生成AI」—こういうテーマの記事がnoteには毎日たくさん投稿される。でも手動で巡回していると、どうしても見逃しが多い。

フォローしているクリエイターの投稿はタイムラインに流れてくるが、フォロー外の良記事は埋もれてしまう。検索すれば出てくるが、毎日それをやるのはコストが高い。

**「気になるテーマのバズ記事が、毎朝Discordに届いていたら最高じゃないか」**

そう思って設計を始めた。

## やったこと

設計フェーズで決めたことを整理する。

### noteからどうデータを取るか

まず調べたのはデータ取得の方法。選択肢は4つあった。

| 方法 | 評価 |
|---|---|
| 公式API | 存在しない |
| RSS | ユーザー別のみ。タグ別・人気記事別のフィードは存在しない |
| 非公式API v3 | スキ数・有料フラグ・フォロワー数まで取れる。採用 |
| スクレイピング | フォールバックとして残す |

非公式API v3（`https://note.com/api/v3/`）はドキュメントこそないが、広く使われており情報も多い。robots.txt で記事・マガジンのクロールは許可されているので、節度あるアクセスなら問題ないと判断した。

主要なエンドポイントはこれだけ：

```
# キーワード検索
GET https://note.com/api/v3/searches?context=note&q={keyword}

# 記事詳細（スキ数・有料フラグ・コメント数）
GET https://note.com/api/v3/notes/{article_id}/

# クリエイター情報（フォロワー数）
GET https://note.com/api/v2/creators/{creator_id}
```

### スコアリングをどう設計するか

「良い記事」の定義が設計の核心だった。

最初は単純な重み付き合計（スキ数×0.3＋コメント数×0.3…）を考えていたが、これには問題がある。フォロワー10万人のクリエイターのスキ数1,000と、フォロワー500人のクリエイターのスキ数1,000は、全然意味が違う。

それを解決するために**2軸設計**にした。

```python
days = max((now - published_at).days + 0.5, 0.5)
engagement_weight = likes + comments * 2

# バズスコア: 1日あたりの勢い
buzz_score = engagement_weight / days

# エンゲージメント率: フォロワー規模比
eng_score = engagement_weight / max(creator_followers, 100)
```

- **buzz_score**：公開からの日数で割ることで「今まさに伸びている記事」を検知する
- **eng_score**：フォロワー数で割ることで「無名クリエイターの良記事」も拾える

有料記事は×1.2のボーナスをかける。有料でも人気ということは、それだけ価値があると判断できるからだ。

### LLMをどう組み込むか

数値スコアだけでは「バズっているけど全然テーマ違う」記事を排除できない。たとえば「生成AI」で検索しても、料理ブログのおまけとして触れているだけの記事が混ざる。

そこでClaude Haikuにテーマ適合性を判定させる。

```python
# 候補20件に対して、テーマ適合スコア 0.0〜1.0 を返させる
system = """
ターゲットテーマ: 生成AI×組織導入・展開、経営×生成AI、AI活用実践
タイトルと冒頭文から適合度を {"theme_score": 0.85} 形式で返してください。
"""
```

最終スコアは `buzz_or_eng × theme_score` で計算し、各軸の上位5件をUnionして最大10件を配信する。

通知文はClaude Sonnetで生成。タイトルと冒頭から「なぜ今注目か」を1〜2文にまとめる。

### 定期実行基盤の選定

ここが意外と悩んだ。選択肢を並べてみると：

| 選択肢 | 実行時間上限 | 言語 | コスト |
|---|---|---|---|
| **GitHub Actions** | **6時間** | **何でも可** | **無料（Public）** |
| Cloudflare Workers | 30秒（無料）| JS/TSのみ | 無料 |
| Vercel | 10秒（無料）| JS/TS主体 | 無料〜$20/月 |
| Google Apps Script | 6分/回 | JSのみ | 無料 |
| Google Colab | セッション依存 | Python | 自動化非対応 |

このエージェントは rate limit ディレイを入れながら20件のAPIフェッチをするので、実行に1〜2分かかる見込み。Cloudflare Workers（30秒）・Vercel（10秒）は即アウト。GASはJS縛りがある上に6分制限も薄い。Colabは自動化前提でない。

**GitHub Actions（Public リポジトリ）が唯一の現実解だった。**

Publicにすることへの懸念もあるかもしれないが、APIキーや Webhook URL は GitHub Secrets に入れるので、コードを公開しても漏れない。

月180円程度で回せる見込み。

| 項目 | 月額 |
|---|---|

| Claude Haiku（テーマ判定 × 約600件/月）| 約45円 |
| Claude Sonnet（通知文生成 × 約300件/月）| 約135円 |
| **合計** | **約180円** |

Sonnetをすべてに使うと高くなるが、テーマ判定の大量処理はHaikuに任せることでコストを抑えている。

## 意外だった発見

- **noteにはタグ別の人気フィードがない**。RSS はユーザー別のみ。検索APIを使うしかないと気づいたのは調べ始めてから。
- **フォロワー数は記事APIに含まれない**。記事詳細 → クリエイターAPIと2段階のフェッチが必要なのでレート制限設計が重要になる。
- **Vercel Cron の無料枠は1日1回**。「無料でcronできる」という認識は合っているが、頻度制限があることは見落としがちなポイント。

## 現状と次のステップ

設計は固まった。次はコードを書く。

1. `scorer.py`：モックデータで2軸スコアの動作確認
2. `fetcher.py`：API v3 の疎通確認（実際に叩いてみないと分からないフィールドがある）
3. `llm_judge.py`：Haiku のテーマ判定精度を検証
4. 統合して Discord に実際に通知を飛ばす
5. GitHub Actions で定期実行化

まだ「動くもの」はないが、設計の解像度が上がったことで、実装で迷う箇所が大幅に減った。AIとの壁打ちで1日で設計まで終わらせられたのは素直に良かった。
