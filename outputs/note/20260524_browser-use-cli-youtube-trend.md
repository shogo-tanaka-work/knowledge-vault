# YouTube検索トレンドの収集を、Claude CodeとBrowser Use CLIで1コマンドにした話

毎週月曜の朝、コーヒーを淹れたあとに最初にやっていた作業がありました。Googleトレンドの「探す」タブを開き、地域を日本、期間を過去1ヶ月、フィルタをYouTube検索に切り替え、上位50件のキーワードをスプレッドシートに転記する。所要時間は20〜30分。誰かに任せるほどでもない、こういうタスクが一番長く残ります。

今回はこの作業をClaude CodeとBrowser Use CLIで潰した話を書きます。完成形はターミナルで1コマンド叩けば終わる、地味ですが効くやつです。実装の細部に加え、なぜYouTube Data APIではなくブラウザ自動化を選んだのか、Makeやn8nではなくCLIに振り切ったのか、判断の根拠も残しておきます。

## サマリ

- Googleトレンドの「探す」タブでYouTube検索フィルタを掛けた上位50件を、Claude Codeから1コマンドで取得できるようにした
- 構成はBrowser Use CLI（Playwrightベース）とClaude Codeのスキル化のみ。サーバーもn8nもMakeも使っていない
- YouTube Data API v3では`gprop=youtube`相当のトレンド情報が取れず、ブラウザ自動化に切り替えた
- 引っかかったのはアンチボット判定。URLパラメータ直叩きと描画待ち、必要に応じて実Chromeプロファイル使用で抜けた
- 週20〜30分の手作業が10秒程度に圧縮され、収集頻度を週1から日次に上げる余地ができた

## なぜData APIではなくブラウザ自動化なのか

最初に検討したのはYouTube Data API v3です。無料枠は日次10,000ユニットあり、コストは余裕でした。

それでも見送ったのは、欲しい情報の粒度が違ったからです。Data APIで取れるのは「いま人気の動画」、欲しかったのは「人々がYouTubeで何を検索しているか」。前者は結果の集計で、後者は需要の生データ。コンテンツ企画の起点に置きたいのは後者でした。

加えて、Googleトレンドの`gprop=youtube`で取れるYouTube検索の急上昇クエリは、Data APIでは公式提供されていません。2026年5月時点でもShortsと通常動画を区別するフラグすらAPI側に存在しない状況で、検索キーワードの集計に至っては当然ありません。需要シグナルを取りに行くなら、GoogleトレンドのUIを踏むしかありませんでした。

ApifyやBright Dataの有償スクレイパーも選択肢にはあります。本来そちらが正攻法ですが、月に何百クエリも回すわけでもなく、個人運用の規模感では過剰でした。組織で回す段階になったら寄せる前提です。

## Browser Use CLIを選んだ理由

ブラウザ自動化のツールはいくつか選択肢があります。素のPlaywright、Puppeteer、Browserbaseなどのクラウドブラウザ、Makeやn8nのシナリオ。どれも動きはします。

Browser Use CLIに落ち着いたのは、Claude Codeとの相性が決め手でした。`browser-use open`・`get title`・`eval`・`close`という単純なコマンド体系で、Bashツールからそのまま叩けます。スクリプトファイルに保存せず、対話の中でターミナルから直接ブラウザを操作させられる軽さがあります。

Makeやn8nを見送ったのは、運用の起点が必ずブラウザ越しのダッシュボードになるからです。ターミナルから1コマンドで叩きたい今回の用途には、可視化のメリットより実行経路の遠さが目立ちました。可視化の恩恵が出るのは複数人で運用する段階で、一人で回すうちはCLIのほうが速い、というだけの話です。

クラウドブラウザを使わなかったのはコスト面で、Browser Use CLIは手元のPlaywrightを呼ぶだけで追加課金ゼロでした。

## できあがったフロー

最終的にやっていることは拍子抜けするほど単純です。

```
1. Claude Codeに日本語1行投げる
   例:「Googleトレンドで YouTube検索の上位50件取って」
2. browser-use-googletrendsスキルが発動
3. browser-use openで「探す」タブをYouTube・日本・過去1ヶ月で開く
4. 12秒待機（JS描画完了まで）
5. browser-use get titleで確認、evalでテキスト取得
6. 50件未満ならスクロールして再取得
7. Claude Codeが整形して上位50件をマークダウンで返す
8. browser-use closeで終了
```

URLは決め打ちで以下を使います。

```
https://trends.google.co.jp/trends/explore?date=today%201-m&geo=JP&gprop=youtube&hl=ja
```

`date=today 1-m`が過去1ヶ月、`geo=JP`が日本、`gprop=youtube`がYouTube検索フィルタ、`hl=ja`が日本語UI。URLパラメータを正しく指定しておけば、UIを手動でいじる必要がありません。期間を週次に変えたければ`now 7-d`に切り替えるだけです。

スキル化したことで、自分は中身を全部覚えていなくてよくなりました。evalのクォートのエスケープや待機秒数の細部はSKILL.mdに書いてあり、Claude Codeがそれを読みにいって実行してくれます。

## ハマったところ：取得結果が空になる

最初の数回、`browser-use get text`を叩いても結果が空でした。`open`の直後に取得していたのが原因です。

GoogleトレンドはJavaScriptで描画されるタイプで、URLを開いた瞬間にはDOMが空です。Playwright側はネットワークがアイドルになった時点で読み込み完了と判断しますが、Googleトレンドのフロントは継続的に裏で通信が走るらしく、表面的にはアイドルにならない時間帯があります。

対策は単純で、12秒のsleepを挟みました。`browser-use state`での状態確認だけでは不十分で、状態フラグは完了でも肝心のキーワードリストが描画されていない中途半端な状態が起きます。5秒・7秒・10秒で試して、12秒が安定する境界でした。秒数を切り詰めるより、安全側に倒して再実行コストをゼロにするほうが結局速いという判断です。

## ハマったところ：アンチボット判定

短時間に何度も同じURLを叩いて検証していると、ある回から403が返ってきます。CAPTCHAも無しに静かに403を返してくる、地味で気付きにくい遮断です。

対策は二段で用意しました。

ひとつめは、検証時のリクエスト間隔を空けること。連続で叩いていたところを、1回ごとに数十秒のインターバルを入れる運用に変えました。本番で日次1回しか叩かないなら関係ない話ですが、検証中の試行錯誤には効きます。

ふたつめは、実Chromeプロファイルを拝借する方法です。`--profile`オプションでローカルのChromeユーザープロファイルパスを渡すと、Cookie・キャッシュ・拡張機能をそのまま使ったセッションが起きます。日常的にログインしているChromeの認証状態を引き継ぐので、ボット判定をすり抜けやすくなります。Macなら`~/Library/Application Support/Google/Chrome/Default`、Windowsなら`%LOCALAPPDATA%\Google\Chrome\User Data\Default`あたりです。

ただしChrome本体が起動中だとプロファイルがロックされて起動しません。検証時にうっかりChromeを開いたまま叩いて10分悩んだのは恥ずかしい話です。通常運用では生プロファイルは使わず、ブロックされたときだけ切り替える二段構えにしています。

## 取得結果はLLMに整形させる

`eval "document.body.innerText"`で返ってくるのはページ全体の可視テキストで、トレンド上位50件以外にヘッダーやサイドバーの文字列も混ざります。

ここの整形はClaude Codeに任せました。InnerTextの塊を渡して「上位50件をマークダウンの順序付きリストで」と頼めば、それなりに賢くやってくれます。順位を取り違えていないか、関連クエリと急上昇クエリを混在させていないかを目視で確認するくらいで、整形ロジックを自前で書く必要は無くなりました。

伝統的な自動化（n8nやMake）だと、整形ステップに正規表現やJSONパスを書く前提なので、DOM構造が変わるたびに保守が発生します。LLMに整形を委ねると構造が多少変わっても文脈で吸収してくれるので、運用が楽です。エージェント駆動の自動化に手を出して一番強く感じたのがこの差でした。

精度の保証は落ちます。LLMが抽出を間違える可能性はゼロにはなりません。ただ、手作業も抜け漏れがゼロだったわけではなく、トレードオフとしては許容範囲と判断しています。本番でクライアントに数字として提出するなら、別途バリデーション層が要ります。

## スキル化したことで得たもの

`browser-use-googletrends`という名前でSKILL.mdとreference.mdに手順をまとめました。メリットは三つあります。

ひとつ、指示が短くなる。「Googleトレンドで YouTubeの上位50件取って」と書くだけで、URLパラメータ・待機秒数・スクロール・フォールバックまで自動で実行されます。

ふたつ、再現性が出る。試行錯誤の末に採用した部分だけがスキルに残るので、半年後の自分が迷いません。

みっつ、別環境へ持っていきやすい。スキルファイルはMarkdown一枚で、別マシンにBrowser Use CLIさえ入れれば同じ動きが再現できます。複数クライアントで展開する想定があるなら効きます。

## 旧手法との比較

Makeで同じことをやろうとした過去があるので、ざっくり比較を残します。

Makeで生URLを叩く構成は、JSレンダリング前提のページなので生HTMLに欲しいデータが入りません。ヘッドレスブラウザを別途呼ぶ必要があり、BrowserlessやScrapingBeeのような有料サービスを噛ませることになります。月額の固定費が乗り、シナリオの中継ポイントも増えます。

n8nセルフホストならコストは抑えられますが、ヘッドレスブラウザを内部で回す構成は実装の手間が大きく、そこで詰まりました。

Browser Use CLIとClaude Codeなら外部サービスを噛ませずに同じことができ、ローカルPCで完結します。代わりにPCが起動していないと動かないという制約はあります。スケジュール実行を組みたければ、ヘッドレスブラウザを別途立てる必要があり、そこは別設計です。

## どれくらい速くなったか

手作業は週1で20〜30分、年間20時間前後でした。仕組み化後はClaude Codeに1行投げて結果が返るまで10〜15秒、スプレッドシート転記もマークダウン貼り付けで1分かかりません。年間で16時間以上は浮きます。

数字としては派手ではないですが、地味な定例作業がなくなった心理的な軽さは数値以上です。「あ、月曜の朝のあれ、もう要らないんだ」と思える瞬間が、業務自動化のいちばん効くタイミングだと思っています。

加えて、収集頻度を上げる選択肢が手に入りました。手作業のままだと週1が限界ですが、コマンド1発になれば日次にしても負担はほぼ変わりません。日次なら急上昇キーワードの立ち上がりを2〜3日早く拾える可能性があり、コンテンツ企画の起点を早めるという二次効果のほうが、たぶんいちばん大きな収穫でした。

## 弱点も書いておく

中間まとめなので、現状の弱点も並べます。

**安定性は完全ではない**: 12秒待機で大半は通るものの、稀に描画が遅れて取得結果が薄い回があります。Claude Code側に「10件未満なら再取得」のガードを入れていますが、根本的にはGoogleトレンド側の挙動次第です。

**構造変更への耐性**: `document.body.innerText`を素直に取りにいく方式は、UIが大きく変わったら効かなくなります。半年に1回くらいは点検が必要だと割り切っています。

**スケールアップは設計し直し**: 複数地域・複数期間を同時に取りたくなった段階で、今の構成だと逐次実行になります。`--session`で並列化はできますが、アンチボット判定のリスクが上がります。本格運用なら有償スクレイパーへの移行が筋です。

**法令・利用規約**: スクレイピングを商用利用するなら利用規約の確認が要ります。今回は社内調査目的に閉じた利用なのでグレーゾーンの中で運用していますが、公開コンテンツの裏付けや有償サービスへの組み込みでは、BigQuery経由のGoogle Trends公式コネクタや有償データセットへ寄せたほうが安全です。

## 横展開

同じBrowser Use CLIのスキル構造は他のシナリオにも転用できます。SNSキャンペーンの進行集計、競合LPのレイアウト変更検知、求人サイトのポジション数推移、決算情報の特定数値抽出など、JSレンダリング前提のサイトに対する定型テキスト取得ならどれにでも応用できます。

ブラウザ自動化とLLM整形の組み合わせは、ここ1〜2年で一気に現実的になった選択肢です。Playwright単体で書いていた頃の苦労を思い出すと、CLI 1行で済むようになったのは率直に驚きでした。自動化の最後のひと押しがLLMになったことで、これまで諦めていたタスクの境界が一段押し下げられている感覚があります。

## まとめ

GoogleトレンドのYouTube検索キーワード取得をBrowser Use CLIとClaude Codeのスキル化で1コマンドに圧縮した、それだけの話でした。仕組みは単純ですが、地味な定例作業が消えた効きは大きく、収集頻度を上げる二次効果のほうがおまけとして大きかった、というのが正直な感想です。

Make・n8n・有償スクレイパーと比べると、Browser Use CLIとClaude Codeはローカル完結のぶんスケーラビリティと安定性で劣ります。逆に、個人運用やスモールチームでは追加課金ゼロ・即時起動の強みが効きます。組織で回す段階になったらサーバーレス化や有償移行を検討する、それまでの中間ステップとしてはちょうどよい構成です。

次の課題はスケジュール実行の常設化で、いまは手元で叩く運用です。日次自動化にはPCが常時起動している必要があり、Raspberry Piに寄せるか軽量VPSでPlaywrightを回すか、別途検証する予定です。進捗はまた書きます。
