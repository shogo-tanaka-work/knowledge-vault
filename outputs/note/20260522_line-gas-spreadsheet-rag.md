# LINEサポートのFAQ対応をAI化してみた｜DifyなしでスプレッドシートとGASだけで作る簡易RAGの話

LINE公式アカウントのチャットサポートを運用していると、同じ質問に何度も答えている自分に気づく瞬間がある。料金は、納期は、キャンセルは——内容の8割は過去に答えた質問の焼き直しだ。

そこで「FAQをAIに一次対応させたい」となるわけだが、ここで多くの人が一度つまずく。チャットボットのSaaSを契約する、AIワークフローツールを噛ませる、専用のサーバーを立てる。どれも動くけれど、運用を引き継ぐ現場の担当者からすると「また新しい管理画面が増えた」になる。

今回はあるクライアント案件で、その「管理画面が増える問題」を避けながらLINEのFAQ対応をAI化する構成を検証した。結論から言うと、運用者が触るのは**Googleスプレッドシート1枚だけ**で済む形に落ち着いた。まだ精度評価の途中ではあるけれど、設計の判断と引っかかったところを中間報告としてまとめておきたい。

## TL;DR

- LINEのFAQ対応を、Dify等のワークフローツールを介さず「LINE + Google Apps Script + スプレッドシート + OpenAI API」だけで組んだ
- ユーザーの自由入力に対し、**埋め込み → 類似度検索 → LLM最終判定**の三段構えで「答えるべき質問か」を判定する簡易RAG構成
- 運用担当者が触るのはスプレッドシートのQ&A列だけ。ベクトル化はスクリプトが裏で自動管理する
- 実機のLINEからの疎通（E2E）までは確認済み。**回答精度の本格評価はこれからのフェーズ**
- GAS特有の制約（HTTPヘッダが読めない等）で妥協した点もあり、そこも正直に書く

## なぜ「Difyあり」をやめたのか

最初の設計は、実はDifyを中継する案だった。LINEのWebhookをDifyの開始ノードで受けて、そこからGASを呼ぶ。AIワークフローを可視化できるし、定番の構成ではある。

ところが詰めていくと、二つ引っかかった。

ひとつは**ユーザー体験**。Dify経由だと回答の導線が「Webの回答画面へ誘導する」形になりやすく、LINEのトーク画面でそのまま会話が完結する通常のチャット体験から外れてしまう。FAQボットなのに、わざわざ別画面に飛ばすのは違う。

もうひとつは**運用負荷**。Difyのノード設定は、作った本人ならいいが、引き継いだ運用担当者にとっては「もうひとつ覚えることが増えた」状態になる。今回のクライアントは、AIの裏側ではなく日々のFAQ更新だけに集中したい。

この二つを天秤にかけて、Difyを外した。構成要素を「LINE / GAS / スプレッドシート / OpenAI」の4つだけに絞り込む。障害が起きたときに見る場所が減るし、運用の窓口がスプレッドシートに一本化される。可視化ツールの便利さよりも、引き継ぎやすさを取ったということだ。

## 作ったものの全体像

仕組み自体はシンプルだ。

```
LINE Messaging API
   │  Webhook（メッセージ受信）
   ▼
Google Apps Script（doPost）
   │  ① 質問を埋め込みベクトルに変換
   │  ② スプレッドシート全行とコサイン類似度を計算
   │  ③ 上位候補をLLMに渡して最終判定
   ▼
LINE 返信API → ユーザーへ回答 or フォールバック文言

スプレッドシート「knowledge」シート ＝ 簡易ベクトルDB
  A:ID  B:質問  C:回答  D:embedding（自動）  E:hash（自動）
```

ユーザーがLINEで質問を送ると、GASがそれを受け取って「スプレッドシートに登録済みのFAQのうち、意味的に一番近いものはどれか」を判定し、十分に近ければその回答を、近いものがなければ「お答えできません」系の固定文言を返す。

ポイントは、スプレッドシートを**簡易的なベクトルDB**として使っているところだ。専用のベクトルデータベースを立てるほどの規模ではない案件なので、Q&Aと一緒に「embedding（質問をベクトル化した数値列）」を同じシートに持たせている。

## キーワード一致ではなく「意味の近さ」で探す

FAQ検索でいちばん素朴な実装は、キーワードの部分一致だ。「料金」という単語が入っていたら料金のFAQを返す、という方式。

これは案外もろい。「いくらかかりますか」「費用感を教えて」「予算ってどれくらい見ておけば」——どれも料金の質問なのに、「料金」という単語はひとつも入っていない。キーワード一致では拾えない。

そこで使ったのが**埋め込み（embedding）**だ。文章をAIが「意味」を保ったまま数値の並びに変換する技術で、意味が近い文章どうしはベクトルとしても近くなる。「いくらかかりますか」と「料金はおいくらですか」は、単語が違ってもベクトル空間では隣り合う。

この「ベクトルの近さで関連文書を引き当て、その内容をもとにAIが応答する」仕組みを、一般にRAG（検索拡張生成）と呼ぶ。今回はそのRAGを、専用基盤を使わずスプレッドシート上で簡易的に再現した、という位置づけになる。

## 三段構えで「答えるべき質問か」を判定する

ただ、意味が近いものを引っ張ってくるだけだと、もうひとつ問題が残る。**「近いけど、実は答えになっていない質問」をどう弾くか**だ。

FAQに無いことを聞かれたとき、それでも「いちばん近いFAQ」は必ず存在してしまう。近さのスコアがそこそこあるからといって機械的に返すと、見当違いの回答を堂々と送りつけることになる。これはサポート品質としていちばん避けたい。

そこで判定を三段構えにした。

1. **埋め込み** — ユーザーの質問を1536次元のベクトルに変換する
2. **類似度検索** — スプレッドシートの全行とコサイン類似度を計算し、上位5件の候補を抜き出す
3. **LLM最終判定** — 最大類似度が一定値（今回は0.7）以上で、かつLLMが「この候補は質問に関連している」と判定したときだけ回答を返す

数値スコアという機械的なフィルタと、LLMという意味的なフィルタを直列に置く。どちらかが「これは違う」と言えば、回答せずにフォールバック文言に切り替える。**ボットが堂々と間違えるより、答えられないと正直に言うほうがマシ**、という思想だ。

LLMの判定には`gpt-5-mini`を使い、出力をJSON形式に固定した。`relevant`（関連あるか）と`index`（どの候補か）の2フィールドだけを返させる。出力フォーマットを固定しておくと、LLMが余計な前置きや言い訳を返してこなくなり、後続の処理が安定する。地味だが効いた工夫だ。

応答時間は、埋め込みが0.3秒前後、LLM判定が1〜3秒程度。LINEの返信に使えるトークンには有効時間があるけれど、そこに対しては十分に余裕をもって返せている。

## 運用者が触るのは「スプレッドシート3列」だけ

今回いちばんこだわったのが、運用担当者の作業範囲だ。

`knowledge`シートは5列ある。A列がID、B列が質問、C列が回答、D列がembedding、E列がhash。このうち**運用担当者が触るのはA〜Cの3列だけ**。D列のembeddingとE列のhashは、スクリプトが裏で勝手に管理する。

担当者の作業はこうなる。新しいFAQが増えたら、シートに行を足して質問と回答を書く。それだけ。ベクトル化のことも、ハッシュのことも、APIのことも知らなくていい。

ベクトル化のきっかけは二つ用意した。ひとつはスプレッドシートのカスタムメニュー（「チャットボット」メニューから「インデックス再構築」を選ぶ）。もうひとつは毎日決まった時刻に走る自動トリガ。担当者が朝にFAQを編集しておけば、その晩のうちに自動で反映される。「保存ボタンを押す」感覚すら要らない。

ここまで運用を単純化できたのは、Difyを外して窓口をスプレッドシートに一本化すると決めた最初の判断が効いている。

## 全件を毎回ベクトル化しない——差分検知の話

FAQをひとつ直すたびに全件をベクトル化し直すのは、無駄が大きい。OpenAIのAPIは使った分だけ課金されるし、件数が増えれば時間もかかる。

そこで**差分検知**を入れた。各行について「質問＋回答」の文字列からSHA-256ハッシュを計算し、E列に保存しておく。次にインデックスを再構築するとき、いまのハッシュと保存済みハッシュを比べ、**一致する行はスキップ、変わった行（と、まだベクトル化されていない行）だけ**をベクトル化する。

これで、20数件あるFAQのうち1件だけ直しても、APIに送られるのはその1件分だけ。再実行のコストはほぼゼロに抑えられた。「データの更新はリアルタイムでなくていい」という前提を最初に確定できたので、日次トリガ＋差分検知という割り切った設計に踏み込めた。

## 正直に書く、引っかかったところ

ここまで読むときれいに収まったように見えるけれど、妥協した点もある。中間報告なので、そこも書いておく。

### LINE公式の署名検証が、GASでは実装できなかった

LINEのWebhookには、リクエストが本当にLINEから来たものか検証する仕組みがある（`X-Line-Signature`というHTTPヘッダを使う）。本来はこれを使いたい。

ところがGASのWebアプリには制約があって、`doPost`という受け口の関数は**リクエストのHTTPヘッダを読めない**。本文（POSTデータ）は取れるのに、ヘッダは取れない。つまりLINE公式の署名検証はそもそも実装不可能だった。

代替として、Webhook URLの末尾に秘密の文字列をクエリパラメータとして付け、受信側でそれが一致するかを見る簡易認証にした。正規の署名検証ではないので、ここは妥協点だとはっきり言える。ただし、HMAC署名を検証するロジック自体はコード内に「使わないけど残してある」状態にしてあり、将来GAS以外の基盤（Cloud Run等）に移すときにそのまま流用できるようにした。

### キャッシュに上限がある

GASのキャッシュ機能には1項目あたり100KBという上限がある。ベクトルの一覧をキャッシュに載せて高速化しているが、FAQ件数が増えるとこの上限を超える。超えた場合は自動でスプレッドシートを直接読みにいくフォールバックを入れたので動作は止まらないけれど、件数が数百〜数千件規模に育ったときにどう振る舞うかは、まだ実測できていない。

### 類似度の閾値が、まだ「仮」

「最大類似度0.7以上なら回答候補とみなす」の0.7という数字。今は現行のFAQ約20数件で動かしているが、件数が増えればスコアの分布も変わる。今の0.7が妥当かどうかは、正直まだ判断しきれていない。データが揃った段階で再調整する前提のパラメータだ。

## いまどこまで来ているか

現時点で確認できているのは、**実機のLINEからメッセージを送って、登録済みの質問には正しい回答が、無関係な質問にはフォールバック文言が返る**という一連の動作（E2E）まで。差分検知も、不正なリクエストの破棄も、メニュー操作も自動トリガも、想定どおり動いている。

一方で、**回答精度の本格的な評価はこれからのフェーズ**だ。クライアントの既存マニュアルをQ&A形式に整理し直したうえで、「実際の問い合わせに対してどれくらいの精度で正しく答えられるか」を測る。ここが終わって初めて、このボットを本番運用に乗せられる。

正直に言えば、簡易RAGの「作る」部分は今回それほど大変ではなかった。本当に難しいのは、**マニュアルをどの粒度のQ&Aに分解するか**——つまりFAQの作り方そのものだ。質問の切り方が雑だと、どんなに検索の仕組みが良くても精度は出ない。ここはまだ属人的で、標準化が次の宿題になっている。

## この構成が向いている場面

今回の「スプレッドシート＋GAS＋LINE」構成は、こういうケースにそのまま応用が利く。

- LINE公式アカウントでの問い合わせ一次対応
- 社内向けのFAQボット（経費精算は、申請フローは、といった定番質問の受け）
- 数十〜数百件規模で、更新がリアルタイムでなくてよいナレッジ

逆に、ナレッジが数千件を超える、回答をリアルタイムで反映したい、といった要件になると、専用のベクトルDBやサーバー構成に移行したほうがいい。今回の構成は「専用基盤を立てるほどではないが、キーワード検索では物足りない」というちょうど中間の規模に効く。

ツールを足し算する前に、「運用する人がいちばん触りやすい場所はどこか」から逆算する。今回の検証でいちばん持ち帰った学びは、技術選定よりもむしろそこだった。

## まとめ・次にやること

DifyのようなAIワークフローツールは便利だが、便利さと引き換えに「運用者が覚えること」が増える。今回はそれを外し、運用窓口をスプレッドシート1枚に絞った簡易RAGをLINE上で動かすところまで来た。

次のフェーズでやることは決まっている。既存マニュアルのQ&A化、回答精度の定量評価、件数を増やしたときのスケール耐性の実測、そして類似度閾値の再調整。進捗はまた別の記事で報告したい。
