# LINEサポートのFAQ対応をAI化してみた｜DifyなしでスプレッドシートとGASだけで作る簡易RAGの話

LINE公式アカウントのチャットサポートを運用していると、料金は、納期は、キャンセルは、と同じ質問に何度も答えている自分に気づきます。問い合わせの8割は過去の焼き直しです。

今回はクライアント案件で、LINEのFAQ対応をAI化する構成を検証しました。落とし所として、運用者が触るのは**Googleスプレッドシート1枚だけ**になっています。回答精度の本格評価はこれからですが、設計判断と引っかかった点を中間報告としてまとめます。

## 先に要点

- LINEのFAQ対応を、Dify等のワークフローツールを介さず、LINE・Google Apps Script・スプレッドシート・OpenAI APIの4要素だけで組みました
- ユーザーの自由入力に対し、**埋め込み → 類似度検索 → LLM最終判定**の三段構えで、回答すべき質問かどうかを判定する簡易RAG構成です
- 運用担当者が触るのはスプレッドシートのQ&A列だけ。ベクトル化はスクリプトが裏で自動管理します
- 実機LINEからの疎通（E2E）までは確認済みで、**回答精度の本格評価は次フェーズ**です
- GAS特有の制約（HTTPヘッダが読めない等）で妥協した点もあり、そこも正直に書きます

## なぜDifyを使うのをやめたのか

最初はDifyを中継する設計で進めていました。LINEのWebhookをDifyの開始ノードで受け、そこからGASを呼ぶ構成です。AIワークフローを可視化できる定番の構成ではあります。

ただ、詰めていくうちに二点で引っかかりました。

ひとつはユーザー体験です。Dify経由だと回答の導線がWebの回答画面側に寄りやすく、LINEのトーク画面で会話が完結する通常のチャット体験から外れてしまいます。FAQボットなのに別画面に飛ばすのは違うと判断しました。

もうひとつは運用負荷です。Difyのノード設定は、引き継いだ運用担当者にとっては覚えることがひとつ増えた状態になります。今回のクライアントが集中したいのはAIの裏側ではなく日々のFAQ更新のほうなので、可視化ツールの便利さよりも引き継ぎやすさを取りました。

結果、構成要素はLINE・GAS・スプレッドシート・OpenAIの4つだけに絞り、運用窓口をスプレッドシートに一本化しています。

## 作ったものの全体像

```
LINE Messaging API
   │  Webhook（メッセージ受信）
   ▼
Google Apps Script（doPost）
   │  ① 質問を埋め込みベクトルに変換
   │  ② スプレッドシート全行とコサイン類似度を計算
   │  ③ 上位候補をLLMに渡して最終判定
   ▼
LINE 返信API → ユーザーへ回答 or フォールバック文言

スプレッドシート knowledge シート ＝ 簡易ベクトルDB
  A:ID  B:質問  C:回答  D:embedding（自動）  E:hash（自動）
```

ユーザーの質問をGASが受け取り、スプレッドシート上のFAQで意味的にいちばん近いものを探します。十分に近ければその回答を、近いものがなければお答えできません系の固定文言を返します。

工夫したのは、スプレッドシートを**簡易ベクトルDB**として使っている点です。専用ベクトルデータベースを立てるほどの規模ではないため、Q&Aと並べて質問のembedding列を同じシートに持たせています。

## キーワード一致では取りこぼす

「いくらかかりますか」「費用感を教えてほしい」「予算はどれくらい見ておけばいいか」――どれも料金の質問ですが、「料金」という単語は入っていません。部分一致では拾えない領域です。

そこで使ったのが**埋め込み（embedding）**です。文章を意味を保ったまま数値列に変換し、近い意味の文章はベクトルとしても近くに来るという性質を利用します。ベクトルの近さで関連文書を引き、その内容をもとにAIが応答する仕組みを一般にRAG（検索拡張生成）と呼びます。今回はそれを専用基盤を使わずスプレッドシート上で簡易再現した、という位置づけです。

## 三段構えで「答えるべき質問か」を判定する

意味が近いものを引っ張るだけだと、近いけれど実は答えになっていない質問を弾けません。FAQに無いことを聞かれてもそれなりに近いFAQは必ず存在してしまい、機械的に返すと見当違いの回答を堂々と送ることになります。サポート品質としていちばん避けたい事態です。

そこで判定を三段にしました。

1. **埋め込み** — ユーザーの質問を1536次元のベクトルに変換
2. **類似度検索** — 全行とコサイン類似度を計算し、上位5件の候補を抜き出す
3. **LLM最終判定** — 最大類似度が一定値（今回は0.7）以上で、かつLLMが「関連している」と判定したときだけ回答を返す

数値スコアという機械的フィルタと、LLMという意味的フィルタを直列に置く構成です。どちらかが否と判断すればフォールバック文言に切り替えます。ボットが堂々と間違えるくらいなら、答えられないと正直に言うほうがよい、という思想です。

LLM判定には `gpt-5-mini` を使い、出力をJSONに固定しました。`relevant`（関連の有無）と `index`（候補番号）の2フィールドのみを返させます。出力フォーマットを固定すると余計な前置きが入らず、後続処理が安定します。地味ですが効きました。

応答時間は埋め込みが0.3秒前後、LLM判定が1〜3秒程度で、LINEの返信トークンの有効時間に対して十分余裕があります。

## 運用者が触るのはスプレッドシート3列だけ

今回いちばんこだわった点が、運用担当者の作業範囲です。

`knowledge` シートの5列のうち、**運用担当者が触るのはA〜Cの3列（ID・質問・回答）だけ**で済みます。D列のembeddingとE列のhashはスクリプトが裏で管理するため、新しいFAQが増えたら行を足して質問と回答を書くだけです。

ベクトル化のきっかけは二つ用意しました。スプレッドシートのカスタムメニューから「インデックス再構築」を選ぶ手動操作と、毎日決まった時刻に走る自動トリガです。担当者が朝にFAQを編集しておけば、その晩のうちに自動で反映されます。

## 全件を毎回ベクトル化しない（差分検知）

FAQを一つ直すたびに全件再ベクトル化するのは無駄です。OpenAI APIは使った分だけ課金されますし、件数が増えれば時間もかかります。

そこで**差分検知**を入れました。各行について質問と回答をつないだ文字列のSHA-256ハッシュをE列に保存し、再構築時に保存済みハッシュと比較。一致する行はスキップし、変わった行と未ベクトル化の行だけを処理します。20数件のうち1件だけ直した場合、APIに送られるのもその1件分だけです。

データ更新がリアルタイムでなくてよいという前提を最初に確定できたので、日次トリガと差分検知という割り切った設計に踏み込めました。

## 裏で何が起きたかを後から追えるようにする

チャットボット運用でいちばん困るのは「さっきの質問になぜ変な答えが返ったのか」に答えられないことです。LINEのトーク画面には結果しか残らず、内部判定は見えません。

そこで処理の節目ごとにスプレッドシートのログシートにイベント（受信・処理・回答・フォールバック）を追記し、スコアや判定結果もJSONで残しています。ログの置き場をスプレッドシートにしたのは、運用担当者がGASのクラウドログを見にいくのは現実的でないためです。同じファイルの別シートを開くだけで「今日はどれに答えられて、どれに答えられなかったか」を眺められます。

これは検証初期に組み込んでおいて正解でした。LINEから実際に飛んでくるリクエスト形式はドキュメントだけでは分からない部分があり、ログを見ながら確認できたのが大きかったです。

## 正直に書く、引っかかったところ

中間報告なので妥協点も書いておきます。

### LINE公式の署名検証がGASでは実装できなかった

LINEのWebhookには `X-Line-Signature` ヘッダによる署名検証の仕組みがあります。本来はこれを使いたかったのですが、GASのWebアプリでは `doPost` がHTTPヘッダを読めません。本文は取れてもヘッダは取れず、署名検証は実装不可能でした。

代わりにWebhook URL末尾に秘密文字列をクエリパラメータで付け、それが一致するかを見る簡易認証にしています。正規の署名検証ではなく、明確な妥協点です。ただしHMAC署名のロジック自体はコードに残してあり、将来Cloud Run等に移すときそのまま流用できます。

### キャッシュに上限がある

GASのキャッシュは1項目あたり100KBが上限です。ベクトル一覧をキャッシュに載せて高速化していますが、FAQが増えると上限を超えます。超えた場合は自動でスプレッドシートを直接読みにいくフォールバックを入れたので動作は止まりませんが、数百〜数千件規模での挙動はまだ実測していません。

### 類似度の閾値0.7はまだ仮

最大類似度が0.7以上なら回答候補とみなす、この0.7はまだ仮の値です。現状20数件で動かしていますが、件数が増えればスコア分布も変わります。データが揃った段階で再調整する前提のパラメータです。

## いまどこまで来ているか

実機LINEからメッセージを送り、登録済みの質問には正しい回答、無関係な質問にはフォールバック文言が返るというE2Eの動作までは確認できています。差分検知・不正リクエストの破棄・メニュー操作・自動トリガもいずれも想定どおりです。

残るは**回答精度の本格評価**で、クライアントの既存マニュアルをQ&A形式に整理し直したうえで実際の問い合わせにどれだけ正しく答えられるかを測るフェーズに入ります。

正直に言うと、簡易RAGを作る部分は今回それほど大変ではありませんでした。本当に難しいのはその先で、元のマニュアルをどの粒度のQ&Aに切り分けるかというFAQ設計のほうです。質問の切り方が雑だと検索の仕組みがどれだけ良くても精度は出ません。ここはまだ属人的で、標準化が次の宿題になっています。

## コストはどれくらいか

お金がかかるのはOpenAI API部分だけです。1問あたり埋め込みとLLM判定の2回が走りますが、現状20数件規模ではデータ量も小さく、検証期間中は手持ちのAPI枠で十分まかなえています。スプレッドシート・GAS・LINE Messaging APIはいずれも今回の使い方なら無料の範囲です。

問い合わせ件数やFAQ規模が大きくなればAPIコストは増えますが、その段階に来たら専用基盤への移行を検討するタイミングでもあります。小さく始めて伸びてから判断する、という方針に振り切った構成です。

## この構成が向いている場面

今回の構成は次のようなケースに応用できます。

- LINE公式アカウントの問い合わせ一次対応
- 社内向けFAQボット（経費精算・申請フロー等の定番質問）
- 数十〜数百件規模で、更新がリアルタイムでなくてよいナレッジ

逆にナレッジが数千件を超える、回答をリアルタイムで反映したい、といった要件では専用ベクトルDBやサーバー構成に移行したほうがよいです。今回の構成が効くのは、専用基盤を立てるほどではないがキーワード検索では物足りない、という中間規模です。

## まとめ・次にやること

DifyのようなAIワークフローツールは便利ですが、便利さと引き換えに運用者の学習コストが増えます。今回はそこを外し、運用窓口をスプレッドシート1枚に絞った簡易RAGをLINE上で動かすところまで来ました。

次は既存マニュアルのQ&A化、回答精度の定量評価、件数を増やしたときのスケール耐性の実測、そして類似度閾値の再調整です。進捗はまた別記事で報告します。
