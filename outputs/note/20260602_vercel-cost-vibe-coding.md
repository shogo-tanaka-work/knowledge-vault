# バイブコーディングでVercelに載せて終わり、にしていませんか。$20のはずが$132だった請求書の話

AIに指示するだけでWebアプリが組み上がる時代になりました。コードをほとんど書かずに動くものを作る、いわゆるバイブコーディングです。作ったものをそのままVercelに載せれば、数分で世界に公開できます。

この流れを解説している人はたくさんいます。ただ、その先の話をしている人をほとんど見かけません。公開したアプリが、毎月いくらかかるのか。その請求書を開いたことがある人は、エンジニア以外だとかなり少ないはずです。

私のところに届いた請求書は、契約していたプランの6倍を超えていました。

## $20のプランで、$132.62

契約は月$20のProプラン。にもかかわらず、その月の請求は$132.62でした。差額の約$112は、すべてOn-Demandの超過分です。

最初は何かの間違いかと思いました。アクセスが急増したわけでも、重い処理を回したわけでもない。普段どおり開発して、普段どおりデプロイしていただけです。それでも数字は動いていました。

ここで多くの人が「サーバーレスだから常時起動でお金がかかったのでは」と考えます。私もそう疑いました。けれど内訳を見ると、話はまったく別のところにありました。

## 9割は「ビルド」だった

費用の内訳はこうでした。

- Build CPU Minutes：$145.43
- Fast Origin Transfer：$2.53
- Fluid Active CPU：$1.80
- その他：少額

常時起動に近いコストであるFluid Active CPUは、わずか$1.80。今回の犯人ではありません。突出していたのはBuild CPU Minutes、つまりビルドにかかったCPU時間です。全体のおよそ9割をここが占めていました。

アプリの稼働ではなく、アプリを作り直す作業のほうにお金がかかっていた。これが結論でした。

## なぜビルドが積み上がるのか

理由はVercelの親切設計にあります。

VercelはGitHubと連携すると、デフォルトで全ブランチへのプッシュやPR作成のたびに自動でビルドを走らせます。プレビュー用のURLをすぐ出してくれる便利な仕組みです。

ところが開発フローによっては、これが裏目に出ます。私の場合、mainブランチへ細かくPRを出してマージするスタイルでした。featureブランチを切らずに進めることも多かった。つまりコミットして反映するたびに、本番ビルドが一回ずつ走っていたわけです。

```
作業者 → main へPR・マージ
              ↓
         毎回ビルドが走る（そのたびに課金）
```

一回あたりは小さくても、回数が積もれば$145になります。バイブコーディングは試行錯誤の回数が多くなりがちなので、なおさら効いてきます。

## やったこと：ビルドの止血

原因がビルド回数だとわかれば、対策はシンプルです。本番に関係ないビルドを止めればいい。

VercelにはIgnored Build Stepという機能があります。ダッシュボードの各プロジェクトのSettings → Build and Deployment から設定でき、「Only build production」を選ぶと、本番ブランチ以外へのプッシュやPRではビルドが走らなくなります。

- `sales-management-system-dev`：developブランチのみビルド
- `sales-management-system-prod`：mainブランチのみビルド

これでfeatureブランチや未割り当てのブランチに対するビルドは止まりました。

トレードオフはあります。ブランチごとのプレビューURLが出なくなるので、動作確認はdevelopかmainにマージしてから行う運用に変わる。ただ、私の場合はもともとプレビューURLでの確認を運用に組み込んでいなかったので、実害はありませんでした。

なお、設定するときに一点だけ知っておくとよいことがあります。Vercelの公式には「キャンセルされたビルドもデプロイメントクォータにカウントされる」という記載があります。ただしこれはbashコマンドで途中キャンセルする場合の話で、今回採用した「Only build production」はVercel組み込みのオプションなので、ビルド自体が走らない=CPU消費も発生しません。仕組みが違うので混同しないように、というところです。

## もっと効くレバーもある

止血の次に、調べていくと打ち手がいくつもあることがわかりました。同じように請求書に驚いた人のために、効きそうな順に並べておきます。

**ビルドマシンをElasticに切り替える**
ビルドに使うマシンには種類があり、Turbo固定だと$0.126/分かかります。一方、新しいElastic Build Machinesは負荷に応じて自動スケールし、$0.0035/CPU分から。多くのプロジェクトはTurboの性能を使い切らないので、切り替えるだけで半額以下になることもあります。2025年以降の新規チームはElasticがデフォルトですが、以前から使っている場合はTurbo固定のままになっていないか確認する価値があります。

**Spend Managementでハードリミットをかける**
Spend Managementを設定しておくと、50%・75%・100%の段階で通知が飛び、上限に達したら本番デプロイを自動で止められます。新規Proチームは$200がデフォルトの予算になっています。今回のような「気づいたら超過していた」を防ぐ最後の砦です。チェックは数分おきなので、絶対に超えたくない額より少し低めに設定しておくのが安全です。

**モノレポならTurborepo Remote Cache**
複数パッケージをまとめて管理しているなら、Turborepoのリモートキャッシュで変更のないパッケージのビルドをスキップできます。ビルド時間を最大85%削減でき、しかも全プラン無料です。

**用途で乗せ先を分ける**
そもそもNext.js以外のフレームワーク（Astroなど）でLPや静的サイトを作っているなら、Cloudflare Pagesに乗せると帯域は無料、ビルドも実質無制限に近い。管理システムはVercel、軽いサイトはCloudflare、と用途で分けるだけでコスト構造が変わります。

## 作れることと、運用できることは別

ここまでの対応で、ビルド回数は減り、請求の暴走は止まりました。

振り返って思うのは、バイブコーディングで動くものを作れるのと、それを運用し続けられるのは別のスキルだということです。AIは前者をものすごく加速してくれました。けれど、毎月の請求書を開いて、何にいくらかかっているかを読み、止めるべき出費を止める。この後者の部分は、まだ人間が見ていないと誰も見ていない領域になりがちです。

派手ではありません。でも、ここを見られるかどうかが、作って公開できる人と、作ったものを安心して持ち続けられる人の差になっていくと感じています。

バイブコーディングでVercelにアプリを載せている人は、一度だけでいいので請求書のUsageを開いてみてください。Build CPU Minutesの数字が、思っているより大きいかもしれません。
