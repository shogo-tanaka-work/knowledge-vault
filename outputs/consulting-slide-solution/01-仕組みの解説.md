# 01. 仕組みの解説 — なぜこのフローでコンサル品質が出るのか

## 出発点：コンサル風スライドが「AI に作らせると安っぽくなる」問題

PowerPoint 自動生成系の AI ツール（Gamma, Beautiful.ai, etc.）や、ChatGPT に直接 PPTX を作らせるアプローチは、**構造化されすぎた汎用テンプレ**を吐き出すため、

- マッキンゼー / BCG / デロイト的な **「ギチギチに情報が詰まったプロ仕様」** に届かない
- アクションタイトル（数値で結論を断定する1行）を考えてくれない
- 図解の密度（矢印・ボックス・DB図形・番号バッジ等）が表現できない

という問題がある。本ソリューションは、この壁を **3 段階分業** で越える設計にしている。

---

## 中核アイデア：HTML を「設計図」として中継させる

```
[参考スライド画像]
       ↓ (Mode A プロンプト)
[コンサル風スライド生成プロンプト]
       ↓ (画像生成 AI または Claude が解釈)
[HTML/CSS/SVG で構築されたスライド]   ←ココが中間表現
       ↓ (Playwright で PNG 化)
[画像埋込 PPTX（投影向け・見た目完全保持）]
       
       または

[HTML/CSS の設計値を元に]
       ↓ (python-pptx で再構築)
[編集可能 PPTX（ネイティブシェイプ・テキスト）]
```

「Markdown だと表現が浅い、画像生成だと編集できない」という二項対立を、**HTML を中継** することで解決している。

| 中継表現 | 表現密度 | 編集容易性 | PowerPoint 化 |
|---|---|---|---|
| Markdown | 低（線形） | 高 | 弱い |
| 画像（PNG/SVG） | 最高 | 低（再生成必須） | 画像扱い |
| **HTML/CSS/SVG** | **高（自由レイアウト）** | **中（テキスト編集可）** | **2 経路で対応** |

---

## レイヤー1：プロンプトスキル

中核となるのは Claude Code 用の **再利用可能スキル** 2 本。

### consulting-slide-remake
入力: 参考にしたいコンサルスライド画像 1 枚
出力（Mode A）: 戦略コンサル風スライド生成プロンプト
出力（Mode B）: 編集可能 PPTX 化のための工程指示

特徴:
- McKinsey / BCG / NRI / 三菱総研系の配色（ネイビー × ブロンズ × グレー）をデフォルト値として保持
- 「アクションタイトルは具体数値含み・断定形・1 行」「色は 4 色以内・アクセント 3% 以下」「最小 10pt」等の **コンサル品質の制約** をプロンプト内で強制
- KEY TAKEAWAY 必須化、ピラミッド原則準拠

### slide-design-patterns
note アカウント「パワポ研」由来の **約 50 パターン辞書**。
比較・KPIツリー・事業ポートフォリオ・TAM/SAM/SOM 等のテーマ別レイアウト指針を収録。「比較スライド作って」等の依頼で発動。

---

## レイヤー2：HTML/CSS スライド

`slides/*.html` を 1 ファイル 1 スライド構成にしている。

### なぜ HTML か
- **CSS Grid / Flexbox** でマッキンゼー風の 3 カラム＋矢印密集配置が自由
- **inline SVG** で DB シリンダー・書類アイコン・矢印・番号バッジをベクター表現できる
- **テキストはネイティブ HTML** なので diff 管理・部分修正が容易
- **配色は CSS カスタムプロパティ** で一元管理（`assets/style.css`）

### 共有スタイル
`assets/style.css` に、

- ネイビー（`#0b2545` 〜 `#2c5a87`）
- アクセント赤（`#c8102e`）
- デロイト緑（`#4a8b32`）
- 政府系ブロンズ（`#B89456`）
- 16:9 固定枠（1280×720）
- KEY TAKEAWAY 帯
- フッタ・章番号バッジ

など、複数スライド間で一貫性を担保するクラスを定義。

---

## レイヤー3：PowerPoint 出力

2 つの経路を併設している。

### 経路 A: HTML → PNG → 画像埋込 PPTX
- スクリプト: `scripts/01_build_pptx_image_embed.py`
- 内部処理: Playwright + Chrome で各 HTML を 1280×720 @2x の PNG にレンダリング → python-pptx で 16:9 PPTX の各スライドに `add_picture` で全面貼付
- メリット: HTML/CSS/SVG の見た目を **100% 完全再現**
- デメリット: テキスト・図形は **PowerPoint 上で編集不可**（画像扱い）
- ユースケース: **投影用・最終納品**

### 経路 B: HTML 設計値 → ネイティブ再構築 PPTX
- スクリプト: `scripts/02_build_pptx_editable.py`
- 内部処理: HTML の座標・配色値を元に `MSO_SHAPE.RECTANGLE` / `CHEVRON` / `OVAL` / `FLOWCHART_MAGNETIC_DISK` / `FOLDED_CORNER` 等で図形を 1 個ずつ配置、テキストは全てネイティブテキストボックス
- メリット: PowerPoint 上で **クリックすれば全要素編集可能**（文言・色・位置・サイズ）
- デメリット: 元の HTML/SVG と完全一致しない（書類アイコンなどは近似）
- ユースケース: **クライアントに引き渡して微調整させる場合**

### Mode B プロンプトの「守ること」を厳守
- 文字 → 全てネイティブテキストボックス
- 装飾 → 全て PowerPoint シェイプ
- SVG 埋込・OLE 化禁止
- 写真や複雑な実写アイコンに限り PNG 抽出可（今回は使用なし）

`unzip 12-ai-workflow-asis-editable.pptx` で確認すると、`ppt/media/` フォルダが空（画像ゼロ）であることが分かる。これが「編集可能 PPTX」の証拠。

---

## なぜ「Markdown じゃなくて HTML」なのか — 1 行で

> **Markdown は段落が縦に積み上がる構造に縛られるが、HTML/CSS Grid は任意の 2 次元配置・矢印・密度を表現できる**ため、コンサル品質の図解を AI に作らせる際の "中間表現の上限" が大きく違う。

トークンは増えるが、得られる表現密度はそれを上回る。
